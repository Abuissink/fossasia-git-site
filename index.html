Learn Git with FOSSASIA

What is Git? Git is a type of software called a Version Control System. With it, you can manage projects that you, and other people around the world, make and contribute to, without fear of losing code or breaking the project's code. By following along with this tutorial, you'll learn first how to set up Git for projects on your local computer, and build up to using GitHub and being able to contribute to open-source software projects!

But first, you'll need to check you have Git installed on your system. Try typing git in the terminal if you are on Linux or Mac OS X. If you are on Windows, or the git command fails, check the tabs below to learn how to install Git.

Windows Install Instructions

There are a few ways to install Git on Windows. The most official build is available for download on the Git website. Just go to this link. Note that this is a project called Git for Windows, which is separate from Git itself; for more information on it, go to this page.

Another easy way to get Git installed is by installing GitHub for Windows. The installer includes a command line version of Git as well as the GUI. It also works well with Powershell, and sets up solid credential caching and sane CRLF settings. We’ll learn more about those things a little later, but suffice it to say they’re things you want. You can download this from the GitHub for Windows website, at http://windows.github.com

Ubuntu/Debian Linux Install Instructions

You can install Git using apt-get. Just run the command $ sudo apt-get install git-all

Fedora Install Instructions

You can install Git using yum. Just run the command $ sudo yum install git-all

Mac OS X Install Instructions

You can install Git using brew. Just run the command $ brew install git

Other Linux Install Instructions

Gentoo: $ emerge --ask --verbose dev-vcs/git

Arch Linux: $ pacman -S git

openSUSE: $ zypper install git

FreeBSD: $ cd /usr/ports/devel/git
$ make install

Solaris 11 Express: $ pkg install developer/versioning/git

OpenBSD: $ pkg_add git

Get Git up and running

git init

Git init sets up a folder as a Git repo, allowing you to use further git commands. Make sure you are in the folder you want to set up for Git before using this command. A folder containing a project set up for Git is called a repository.

Learn more about Git Init

After the repo has been set up, you can add files into the repo like you would in any other folder. They will not be saved in Git until you commit them to the repo. Before you make the commit however, you need to add files to the commit.

git status

After writing some files, you'll want to add them to the commit. To check which files need to be added to the commit, run git status, which will print out a list of the files and their path that have been changed since the last commit, and need to be added.

Learn more about Git Status

git add

So you've made some changes, and you've used git status to check exactly which files need to be added to the commit. So let's get right to it, run git add [file name] for all the files that have been changed to stage them to the commit. The command git stage [file name] does the same thing as git add, so you can use whichever version you like. You can also add multiple files at the same time by listen the files one after the other, like git add [filename_1] [filename_2].

Learn more about Git Add

git commit

Now that you've staged all your changes, you're ready to commit the changes to Git! The syntax is git commit -m "[insert message here]"

Learn more about Git Commit

git diff

So you've changed files since the last commit, but you're not quite sure what changes you have made. Maybe you accidentally deleted something, and you want to check what has changed. Run git diff [file path/name] to review the changes to the file since the last commit.

Learn more about Git Diff

git log

So say you want to see a log of all the commits that have taken place in the repo. You'd simply run git log, which prints out a list of all previous commits. Notice that each commit has a unique ID number. You may need to press 'q' to exit git log. This command may not seem useful right now, but it will come in handy a lot once you learn more Git.

Learn more about Git Log

Awesome! You now know how to create a local repository, and commit files to it! Next up, we'll learn how to backtrack on changes made in previous commits.

git stash

So sometimes you want to save your changes with Git but don't want to make the step to a whole new commit next time. Easy! Just run git stash and Git will make a record of your changes but not make a commit. To apply the stashed changes later, just run git stash apply. This command comes in handy when you need to save your changes, when you switch branches, for instance (more on this later).

Learn more about Stashing in Git

Backtrack changes in Git

The HEAD Commit / git show

The command git show [commit] is handy if we want to see exactly what a commit changed. The command prints out a log of all the lines added/deleted, and files changed in the specified commit. The HEAD commit is the most recent commit. To see previous commits, use the syntax HEAD~n where n is how far back past the HEAD commit you want to go. For example, the second most recent commit would be HEAD~1.

Learn more about HEAD
Learn more about Git Show

git checkout

So sometimes, you make mistakes. Hey, we all do. What happens if you make mistakes in your code, we know how to use git diff to check exactly what you changed (if you can't remember, no biggie, just check up the page a bit). If you regret making all your changes in a certain file, you can use git checkout HEAD [filename] to restore a certain file to its HEAD version. We'll learn more about git checkout later when we learn more about branches.

Learn more about Git Checkout

git reset

It is good Git practice to keep commits to only one topic. If you've accidentally staged a file for a commit (remember how we learnt that at the beginning?) The command git reset HEAD [filename] resets the file in the staging area to the HEAD version (from the last commit) but keeps the changes in the working directory (as opposed to git checkout, which changes the file in the working directory itself back to the HEAD commit.

git reset SHA

git reset can also be used to reset your project to a happier, stabler commit. Like in the gif above, you'll need the first seven characters of the ID number, or SHA, written in git log. Alternatively, you can use the HEAD~n syntax instead, if you wish. To recap, reset reverts the project to a previous state, and Git forgets about the commits that went after that state.

Learn more about Git Reset

Branching in Git

git branch

Up to this point, we've been working in a single branch, master. If you look on the terminal animations above, you see that (master) appears on all the windows (except for this one :p ). Git allows for the creation of branches, which allow us to experiment with different versions of the project. A branch is essentially another instance of the project running within the project repo, allowing commits to be made that don't affect the master branch.

git branch [branchname]

You can also create branches with git branch. Just add the branch name after the command, as shown in the gif above. Note that you are not automatically switched to the new branch.

Learn more about Git Branch

git checkout

Remember git checkout from earlier? How we used it to reset a file to the HEAD version? This is also the command used to switch branches. If a branch exists, just run git checkout [branchname] to switch to the branch specified.

Go to the first Git Checkout card

git checkout -b

You can actually combine both of the commands above into one easy command. Just run git checkout -b [branchname] and a branch named [branchname] will be created. Notice that it automatically switches to your new branch. Handy!

Learn more about Git Checkout

git merge

So you've made an awesome addition to your code on another branch, and you want to add your changes to the main branch. Switch to the branch you want to pull the changes into, using git checkout [into-branch] and then run git merge [from-branch] where [from-branch] is the branch where you want the changes to come from

Fixing Merge Conflicts

Sometimes, when you try and use git merge it will throw back an error message- something like Merge conflict in [filename] Automatic merge failed; fix conflicts and then commit the result.. Open the file(s) the console gave you, and you'll see something like:
<<<<<<< HEAD
master version of line
=======
otherbranch version of line
>>>>>>> otherbranch
Delete everything except the version you want to keep. Make sure you remove all the Git markings, or Git will still think the merge conflict is present. To finish the merge, just run git commit as normal.

git branch -d

So you've finished with a branch, and merged the changes into the master branch. If you don't need it anymore, run git branch -d [branchname] to delete it.

Using Git with GitHub

GitHub is a website that runs a Git hosting platform, where over 12 million users store and contribute to over 30 million projects. Many open-source software organisations, such as FOSSASIA, store their code in GitHub repos, where anyone can contribute to them. This following section will provide a brief introduction into GitHub, and commands used when dealing with remote repositories. For a way more in-depth guide to GitHub, click the link below.

Learn more about GitHub

git clone

Once, you've signed up/logged into GitHub, you'll want to start contributing to a repo. Once you find one that suits your fancy, click the 'Fork' button to copy the main repo into a personal version listed under your name. All GitHub repos have the url github.com/username/reponame. Once you're in your personal repo, find the HTTPS path in the top right of the repo (see above gif). Make sure you have your personal fork, as you probably won't be able to push your changes to the main repo. Copy that URL, and open up a terminal in the folder where you'd like the repo folder to be located. Run git clone [url] and a folder containing the HEAD commit from the GitHub repo will now exist locally on your computer.

Learn more about Git Clone

git pull

So you've set up the repo from GitHub- great! Now, how do you sync to/from the GitHub server? One half of that is git pull origin [localbranch]. Origin is the remote location, in our case GitHub. Think of git pull origin master as like git merge origin. Git pull is basically a merge, if there is a merge conflict you have to deal with it in the same way too.

Remind me how to deal with merge conflicts
Learn more about Git Pull

git push

Git push is the opposite of git pull and the other half of the sync process. This command takes the commits not registered with the remote repo and registers them there, uploading the new changes to the server. Unlike pull, push requires you to authenticate, just give it your GitHub username and password (assuming the repo is on GitHub!) and the upload should be successful. If not, you may have to use git pull changes from origin first.

Learn more about Git Push

git rebase

Git rebase is probably the most confusing command covered in this tutorial, but it's the most powerful. Rebase is essentially moving a branch from one commit to another. It (like most Git commands) has many different options, but the one shown above, and the one arguably used the most often is git rebase -i HEAD~n, which allows you to squash or flatten several different commits. Once Nano (or whatever your default text editor is) has loaded, you will see a list of all the commits you specified all marked pick. You must keep at least one as pick, but you can change commits listed below a pick to either squash / s or flatten /f to merge their commits with the nearest-up commit marked as pick, keeping and discarding their commit messages respectively. This comes in handy when making a pull request, as it means you only have to deal with one commit in the request.
